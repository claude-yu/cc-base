# Codex Review 收敛策略

基于 v1-v7 实战数据总结的多轮 Codex 审查迭代策略。

## 迭代策略

| 轮次 | 阶段 | 策略 |
|------|------|------|
| 1-3 | 结构性问题 | 严格修复所有 Required Changes |
| 4-5 | 边界问题 | 区分硬性 blocker 和防御性建议 |
| 6+ | 过度优化 | 只修真正 blocker，其余标注后人工接受 |

## 收敛判断

- 连续两轮 Codex 没提出新问题 → 已收敛，可人工接受
- 4+ 轮后仍有"建议"而非"必须修改" → 防御性审查，可接受

## CC 与 Codex 分工

| 问题类型 | 处理者 | 命令/方式 |
|---------|--------|----------|
| Controller/cc-connect 报错 | CC 直接修 | `/修复controller` |
| 脚本、配置、编码、代理、超时 | CC 直接修 | `/修复controller` |
| MD 执行计划、恢复策略 | Codex 审查 | `/计划审查` |
| 分析方案、是否可执行 | Codex 审查 | `/计划审查` |
| Codex REVISE 后改计划 | CC 改 → Codex 再审 | CC + `/计划审查` |

**原则：运维问题 CC 自修，科研/执行风险 Codex 审查。**

## Codex 按需调用（不自动）

**默认不调用 Codex。** CC 独立完成任务。

只在以下场景启动 Codex：
- 用户明确说"让 codex/gpt 审查/检查/分析一下"
- 用户使用 `/计划审查` 命令
- 用户要求独立核查某个结果
- CC 自己判断不了的问题（需先告知用户）

## 计划质询（Grill-Me）

当 Codex 返回 REVISE 且用户不确定如何改进时，可启动 grill-me 模式。

**触发方式**：`/质询计划` 或别名 `grill` / `grillme` / `拷问计划` / `计划质询`

**行为**：
- 无参数时：自动读取最近一次 plan-review run 的 `cc-plan.md` + `codex-review.md`
- 有文本参数时：文本附加为质询上下文

**工作方式**：
1. One-question-at-a-time，每个问题附推荐答案
2. 用户可直接接受推荐答案或修改
3. 质询完成后可重新提交 `/计划审查`

**质询树**（8 个维度）：
1. 成功标准
2. 输入是否就绪
3. 输出是否可验证
4. 不可逆步骤
5. 用户确认点
6. 回退方案
7. Instinct 候选
8. Skill 候选

**当前限制**：不支持 `-RunId` 指定特定 run（v2.2.1 候选）。
